# ADR 0005: YouTube 動画のダウンロードをサポートしない

## ステータス
Accepted — 2026-04-25

## コンテキスト

初期の設計では `yt-dlp` を使って YouTube 動画を直接ダウンロード → 解析する機能を想定していた。
しかし以下の問題がある:

1. **YouTube 利用規約違反**: YouTube の利用規約は「Content を他の方法で取得・複製することを禁止」と明記しており、
   YouTube Data API v3 以外の手段でのダウンロードは明確に ToS 違反
2. **ポートフォリオ価値の毀損**: 採用選考でコードリポジトリを見せた際、
   ToS 違反機能が含まれていると「法令遵守意識の欠如」とネガティブ評価される
3. **法的リスク**: 最悪のケースで損害賠償や IP ブロックのリスク

レビューで明確に指摘されたため、ポリシーを変更する。

## 検討した選択肢

### A. yt-dlp で YouTube DL をサポート（当初案）
- Pros: ユーザーが URL を投げるだけで使える体験
- Cons: ToS 違反、ポートフォリオ価値毀損、リーガルリスク

### B. YouTube は完全に排除
- Pros: リスク最小
- Cons: デモ動画のネタが減る、外部API統合のショーケースが減る

### C. YouTube Data API v3 でメタデータ取得のみ合法利用
- Pros: ToS 準拠、かつ外部API統合の経験は残せる
- Cons: 動画本体の内容解析は別途ユーザーに動画を手元で用意してもらう必要

### D. YouTube は metadata のみ。本体解析はローカル動画/CC動画/自作動画を主軸に
- Pros: 完全合法、外部API統合の経験も残る、実用的
- Cons: Demo の手軽さは落ちる

## 決定

**D を採用**。

具体的な実装方針:
- `POST /api/v1/videos/youtube-metadata` エンドポイントで YouTube Data API v3 経由でメタデータ取得
- 動画本体のインジェスト API（`POST /api/v1/videos`）は YouTube URL を受け取ったら `400 Bad Request` を返す
- yt-dlp 自体はプロジェクトに依存として含めない
- Creative Commons ライセンス動画の URL は受け付ける（サーバーサイドで License 情報を最低限チェック — ベストエフォート）
- デモ用サンプルは `tests/fixtures/sample.mp4`（自作 or CC0）を使う

README とセキュリティセクションに明確に記載。

## 影響・トレードオフ

- デモで「YouTube URL を投げる」体験は提供できない → 手元のサンプル動画を指定する操作が必要
- 将来、YouTube 公式の Partner Program や YouTube Studio API で合法手段ができたら再検討
- 「倫理・法令遵守を考慮した設計」自体がポートフォリオで差別化ポイントになる
  - 面接で「この部分はあえて実装していません」と説明できると好印象

## 再検討トリガー

- YouTube が公式にダウンロードを許可する枠組みを提供した場合
- 同様の問題が別のプラットフォーム（Vimeo等）で発生した場合は同じ方針を適用
